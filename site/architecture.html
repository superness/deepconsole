Architecture

DeepConsole is an Electron app that wraps a chat UI to DeepSeek with an integrated browser the AI can automate. It talks to a sibling backend, abuddi, over local HTTP.

Process model

Main process — main.js

Manages windows and IPC, and runs two internal HTTP servers:

  • Spawns ../abuddi/server.py via uvicorn on port 8000 — the LLM backend.
  • Runs a Browser API server on an OS-assigned port that the backend calls to control the Electron browser window (open, navigate, execute JS, read console logs).

Renderer process — renderer/

All UI logic — chat, session management, streaming SSE events, and the tabs: Browser, Console, JS Runner, Agents, Memory, and Overmind. Lives in app.js, index.html, and style.css.

Preloads

  • preload.js — context bridge for the main window; exposes window.deepconsole.* (llm, agents, abuddi, maestro, memory, browser, overmind).
  • browser-preload.js — context bridge for the AI-controlled browser window; intercepts console.* and forwards entries to the main process buffer.

Data flow for chat

  1. Renderer calls window.deepconsole.llm.chat(sessionId, message) via IPC.
  2. Main process streams Server-Sent Events from http://127.0.0.1:8000/sessions/{id}/chat.
  3. SSE events — token, tool_call, tool_result, ask, done, error — are forwarded to the renderer.
  4. The renderer handles each event type to render streamed text, tool activity, and interactive questions.

The DeepSeek-V4 backend

The backend uses DeepSeek V4, and the model is computed per turn — there is no manual picker. A cheap non-thinking ABUDDI triage pass classifies each turn and emits a complexity_score and a CODE_WORK flag. The work pass then picks:

DimensionRule
Modeldeepseek-v4-pro for code work, else deepseek-v4-flash
Thinkingenabled (effort high, or max when score ≥ 30) once score ≥ 12; routine turns stay non-thinking

reasoning_content is captured and threaded through history so thinking-mode tool-call loops stay valid.

Shared working memory

A unified 3-tier memory system, backed by JSON files in ../abuddi/memory_store/:

TierScopePersistence
Sessiona single conversationephemeral (deleted with the session)
Agentper agent "hat", across sessionspersistent
Metaglobal — all sessions & agentspersistent

The backend exposes REST endpoints at /memory/{tier}/{namespace}/..., the AI has direct memory_* tools, and before every turn the current session + meta memory are injected into the system prompt as live shared state.

Knowledge base

Separate from working memory, a file-based knowledge store holds facts, learnings, and questions the AI curates via knowledge_* tools. It is also injected into every system prompt. Think of it as long-term agent memory, while working memory is the shared 3-tier state machine.

Ports

8000 — the shared abuddi backend. 9200 — the Overmind cross-instance coordinator. The Browser API port is OS-assigned per instance.
